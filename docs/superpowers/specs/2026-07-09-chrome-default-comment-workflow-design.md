# Chrome Default Comment Workflow Design

## Background

The current comment delivery workflow uses the comment-crawler MCP/CDP path as the default browser execution surface. It opens or attaches to a Chrome instance with a remote debugging port, expands comments, captures DOM candidate batches, then sends those batches through AI extraction, normalization, merge, QA, and Excel generation.

The new workflow keeps the existing delivery pipeline and artifact schema, but changes the default browser execution surface to `chrome:control-chrome`. This lets Codex operate the user's logged-in Chrome session through the Chrome extension surface, while preserving the same downstream files consumed by the pipeline.

## Decisions From Grilling

- The change is documentation-only for this iteration.
- Chrome becomes the default main workflow.
- MCP/CDP becomes fallback and legacy debugging guidance.
- Both the `comment-excel-delivery` skill and project docs must be updated.
- The skill stays concise and mandatory; the docs carry longer examples and troubleshooting.
- Outputs must remain fully compatible with the current pipeline.
- Codex automation is preferred for page opening, clicking, scrolling, DOM capture, and tab cleanup.
- Login, CAPTCHA, verification, or platform access checks must pause for user action and must not be bypassed.

## Goals

Make future comment delivery runs follow a Chrome-first workflow:

1. Parse customer requirements into `crawl-tasks.json` and task directories.
2. Use `chrome:control-chrome` to open each task URL in a fresh Chrome tab.
3. Expand comments and replies with Codex-controlled browser interactions.
4. Capture visible DOM candidates into the existing batch artifact shape.
5. Run AI extraction, normalization, task merge, project merge, QA, batch summary, and workbook generation unchanged.
6. Keep MCP/CDP instructions available as fallback for old behavior or MCP-specific debugging.

## Non-Goals

- Do not implement a new CLI in this iteration.
- Do not add or modify MCP tools.
- Do not modify the pipeline Python scripts.
- Do not change `comment-dom-batch-v1`, `ai-comment-extraction-v1`, or normalized comment JSONL shape.
- Do not create Chrome-specific artifact names such as `chrome-dom-batch.json`.
- Do not document or imply any method for bypassing login, CAPTCHA, or platform verification.

## Chrome Data Flow

For each task:

```text
runs/<task_id>/task.json
  -> Chrome opens task.source_url in a fresh tab
  -> Chrome expands visible replies and scrolls
  -> Chrome extracts bounded visible DOM candidates
  -> runs/<task_id>/capture-state.json
  -> runs/<task_id>/batches/<batch_id>/comment-dom-batch.json
  -> AI writes ai-comment-extraction.json
  -> normalizer writes normalized-comments.jsonl
  -> merge_task_batches.py writes task normalized-comments.jsonl
```

The Chrome workflow must use the same batch structure:

```text
output/<project_id>/runs/<task_id>/
  task.json
  capture-state.json
  batches/
    batch_0001/
      comment-dom-batch.json
      ai-comment-extraction.json
      normalized-comments.jsonl
```

Project-level outputs remain unchanged:

```text
all-normalized-comments.jsonl
qa-summary.json
batch-summary.json
delivery.xlsx
```

## Batch Compatibility

Every Chrome-captured batch must satisfy `comment-dom-batch-v1` expectations:

- `schema_version`
- `batch_id`
- `task_id`
- `platform`
- `source_url`
- `captured_at`
- `scroll`
- `state`
- `limits`
- `candidates`

Each candidate should preserve the fields used by the AI prompt and normalizer:

- `candidate_id`
- `candidate_hash`
- `dom_path`
- `role_hint`
- `inner_text`
- `html`
- `nearby_buttons`
- `rect`
- `captured_at`

The docs may show sample extraction code, but the sample must be framed as a way to create compatible batch JSON, not a new schema.

## Browser Safety Rules

The Chrome workflow must document these rules:

- Use a fresh tab per task.
- Do not reuse an old content tab as the task target.
- Verify the current URL and visible page before extracting DOM.
- Pause and ask the user to handle login, CAPTCHA, or platform verification.
- Do not replace authenticated platform access with web search or other sources.
- Close or finalize the task tab after the final capture step.
- If a run is interrupted after files are written, run `resume_comment_project.py` before rerunning and write reruns to the suggested rerun directory.

## MCP Fallback

MCP/CDP remains documented for:

- Reproducing old delivery behavior.
- Debugging MCP candidate capture.
- Running `expand_and_capture_comment_batches` when Chrome extension control is unavailable.
- Comparing Chrome-generated batches against MCP-generated batches.

The fallback section should not compete with the default path. It should explicitly say Chrome is the preferred workflow unless the user requests MCP/CDP or Chrome control is unavailable.

## AI And Pipeline Flow

AI extraction remains unchanged:

1. Read `prompts/comment-candidate-batch-extraction.md`.
2. Read one `comment-dom-batch.json`.
3. Write `ai-comment-extraction.json`.
4. Use input `candidate_id` as `source_chunk_id`.

Normalization remains unchanged:

```bash
node script/normalize-ai-comment-extraction.js \
  --input output/<project_id>/runs/<task_id>/batches/<batch_id>/ai-comment-extraction.json \
  --batch output/<project_id>/runs/<task_id>/batches/<batch_id>/comment-dom-batch.json \
  --task output/<project_id>/runs/<task_id>/task.json \
  --out output/<project_id>/runs/<task_id>/batches/<batch_id>/normalized-comments.jsonl
```

Task and project merge commands remain unchanged:

```bash
python src/pipeline/merge_task_batches.py \
  --task-dir output/<project_id>/runs/<task_id>

python src/pipeline/merge_comment_runs.py \
  --project-dir output/<project_id> \
  --out output/<project_id>/all-normalized-comments.jsonl
```

QA, batch summary, and Excel generation remain unchanged.

## Acceptance Criteria

- Reading the skill makes Chrome the obvious default path for future delivery runs.
- Reading the workflow gives an operator enough steps to run Chrome-first capture and then reuse the existing pipeline.
- Reading the docs gives longer Chrome examples and clearly demotes MCP/CDP to fallback.
- Tests fail if the docs stop saying Chrome is default or if compatible artifact paths disappear.
- No production crawler or pipeline behavior changes in this iteration.
