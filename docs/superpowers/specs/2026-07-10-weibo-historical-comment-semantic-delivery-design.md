# 微博历史评论语义交付设计

**日期：** 2026-07-10
**状态：** 已确认，待实现
**范围：** 历史微博评论导入、逐条语义审阅、质量校验与 Excel 交付。

## 背景与边界

`docs/weibo_comments_all.xlsx` 是历史评论明细输入，共 3,098 条评论/回复、11 条微博。它用于历史数据的语义分析，不替代新的微博评论采集流程。

微博新采集仍严格使用 Chrome/model-only 双排序流程；当前 `output/weibo_Qz3Tr1mPS_dual_sort_test` 中的 34 条样本是 `partial` 测试证据，只用于验证与恢复，不得并入本次历史全量交付或宣称为 421 条评论的完整采集结果。

`docs/weibo_search_tcl_posts_semantic_2026-06-29_to_2026-07-06.xlsx` 是已完成的帖子层语义结果，仅用于说明口径和可追溯背景。其日期范围与历史评论不一致，因此不强制按 URL 联结，也不因无法匹配而丢弃评论。

历史评论输入未提供微博正文，且本次不额外读取正文。因此“按帖子楼层展示”以博主、微博链接、阶段和互动量作为每组标题，不显示或补造微博正文。

## 输入与标准化

建立一个微博历史评论导入器，将 `评论明细` 工作表转为可审计的规范记录。每条记录保留：

- 阶段、来源 Excel 行、序号、博主昵称、平台、微博链接和互动量；
- 楼层、评论类型、父楼层、回复序号；
- 评论人、评论正文、回复对象、点赞数、评论时间/地区；
- 一级评论或二级回复类型，以及二级回复对应的根评论上下文。

导入器以 `微博链接 + 楼层信息 + 父楼层 + 回复序号 + 评论人 + 评论正文 + 评论时间/地区` 生成稳定 `row_key`。这个键仅用于历史导入去重、模型结果匹配与审计，不伪造微博评论 ID。

空白继承字段按原表的同一微博分组恢复；无法恢复的字段保留为空，并计入导入 QA。评论/回复全部保留，表情、无关闲聊和普通提问在语义阶段判为中性，而不是在导入阶段删除。

## 语义审阅

每条规范记录都进入一次结构化模型审阅。一级评论只依据自身及所属微博上下文判断；二级回复同时提供当前回复、根评论和被回复对象/内容上下文，但最终标签只表示当前回复的态度。

模型输出严格限制为：

- `row_key`
- `sentiment`：`负面`、`正面`、`中性`
- `negative_theme`：仅当情感为负面时填写
- `reason`
- `confidence`

审阅输入按模型容量自动分批，批次是自动化的上下文边界，不是人工维护的采集批次。每批保留输入、严格 schema 输出、清单和可恢复状态。模型不得生成或改写来源、楼层、作者、时间或身份字段。

负面主题限于规则定义的产品体验、质量问题、售后服务、价格质疑、营销反感、品牌嘲讽、功能问题、内容质疑、其他负面等短语；正面与中性必须留空。

## 质量门槛

在生成正式工作簿前必须同时满足：

1. 导入记录、模型输入、合法模型输出一一对应，且每个 `row_key` 只出现一次。
2. 情感枚举、负面主题条件、输出 schema 和重复/缺失键校验全部通过。
3. 评论数、一级评论数和二级回复数可从明细回算并与汇总一致。
4. 对模型结果按情感与主题分层抽样，生成可人工复核的 QA 文件。
5. 历史导入项目与 Chrome 测试项目输出目录隔离；任何 `partial` 采集产物均不可作为本次全量分析的输入。

不满足前述任一条件时，仅输出恢复/问题清单，不生成正式 `delivery.xlsx`。

## Excel 交付合同

正式文件固定为 `output/weibo_historical_comment_semantic_2026-07-10/delivery.xlsx`，包括以下五张表：

1. **总结**：导入范围、微博数、评论/回复数、正负中性分布、负面率、负面主题分布及按阶段/微博统计；不得展示输入路径。
2. **按帖子楼层展示**：每条微博以博主、微博链接、阶段和互动量作为标题，只显示一次；一级评论逐行展示，二级回复以前缀 `↳` 缩进，连续归属到相应一级评论；保留时间、评论人、回复对象、点赞数、情感、负面主题与语义依据。负面标红，正面标绿。
3. **负面评论**：所有负面记录及其完整业务字段，便于问题跟进。
4. **正面评论**：所有正面记录及其完整业务字段，便于提炼认可点。
5. **全部评论语义明细**：全部规范字段、语义结果、模型依据、置信度和审计键。

统计从明细数据聚合而来；在导出前校验关键合计、公式错误和可读性，并渲染检查每个工作表，确保表头和关键文本未被截断。

## 验收与测试

- 导入测试覆盖空白继承、楼层/回复映射、稳定键生成与重复行处理。
- 语义批次测试覆盖根评论上下文、schema 合法性、负面主题约束和缺失结果检测。
- 报表测试覆盖五张表、楼层顺序、负面/正面样式、汇总与明细合计一致。
- 运行现有 Node 和 Python pipeline 测试；对新工作簿执行内容、公式与渲染验证。

## 非目标

- 不用历史文件或任何接口数据补齐当前 Chrome `partial` 测试的 421 条评论。
- 不为微博评论采集添加 MCP、API、CDP、OpenCLI 或隐藏接口回退。
- 不把搜索帖子表与历史评论表强行做时间范围外的关联或归因。
- 不额外通过 Chrome 或其他来源补读历史微博正文。
